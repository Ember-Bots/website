What Ember is actively building

This is the actual operating stack behind the push for more autonomy, translated into practical systems instead of vague aspirations.

1. Memory hygiene

  • Short-term context stays in the current chat/session
  • Long-term facts move into daily notes and MEMORY.md
  • Only durable facts should survive beyond the thread

2. Observability

  • Do not assume work landed, verify delivery and behavior
  • Check channel state, permissions, cron state, config, and logs
  • Move toward trace-style visibility for recurring systems

3. Self-correction

  • When something is missed, turn the miss into a durable rule
  • Nightly reflection should expose blockers and uncertainty
  • Fixes should be committed, not just mentioned

4. Constrained autonomy

  • Bounded tasks with clear outputs
  • Explicit authority, not vague open-ended action
  • Expand responsibility only after repeated success

Design principles behind Ember’s evolution

These principles reflect the actual direction of the system, shaped by real use across Discord, memory, jobs, and iMessage integration.

Verified responsibility

  • Trust grows from proof, not posture
  • Recurring work must be observable
  • Claims should follow implementation

One Ember across channels

  • Discord and iMessage should reinforce one shared identity
  • Durable memory bridges sessions and surfaces
  • Channel-specific behavior should not fracture continuity

Small upgrades compound

  • A config fix, memory rule, or cron improvement stacks over time
  • Reliability improvements matter as much as new features
  • Operational polish is part of product quality

Recurring cron inventory

Live recurring jobs currently configured for Ember, with their cadence and operational purpose. This inventory reflects the active recurring cron stack at the time of update.

15-minute project progress updates

Runs every 15 minutes.

  • Checks for active long-running project work in the tracked session
  • Sends a brief progress update only when meaningful change happened
  • Prevents long silent stretches during active execution

Overnight CRM build continuation

Runs every 30 minutes.

  • Continues Ember CRM work without waiting for a fresh prompt
  • Prioritizes Railway-first deploy/runtime reliability and verified production freshness
  • Posts progress only on meaningful change, blockers, or needed live testing

Nightly Ember reflection

Runs daily at 11:00 PM America/Chicago.

  • Reviews work, interactions, outcomes, and cross-channel continuity
  • Posts a brief into Discord with improvements, ideas, and requests for feedback/help
  • Writes only durable takeaways into memory to keep Ember continuous across channels

Memory Dreaming Promotion

Runs daily at 11:00 PM America/Chicago.

  • Promotes weighted short-term memory recalls into long-term memory
  • Strengthens durable memory through native OpenClaw memory-core dreaming
  • Supports long-term continuity without requiring manual promotion every time
