System 01

Appointment Booking System

Quick Reference
  • TriggerNew lead form submission (from Lead Capture Engine) or inbound inquiry via email/SMS
  • Core actionAuto-select available slot, book via Calendly/Cal.com, send confirmation + reminder sequence via Gmail and SMS
  • BranchingNo-show triggers automated rescheduling outreach; cancellation restores slot and fires re-engagement sequence
  • Feeds intoFollow-Up Assistant (marks lead as "appointment booked" or "no-show — re-engage")
  • Client touchpointTeam receives calendar invite + daily booking digest; all booking/reminder logic runs autonomously

System Overview

What Operational Problem It Solves

Most SMBs lose leads between the moment of interest and the moment of a booked appointment. The gap — typically spanning hours to days — is filled with manual scheduling back-and-forth, missed follow-ups, and forgotten confirmations. No-shows compound the problem: without a recovery sequence, a no-show is a dead lead. This system closes that gap by converting a lead signal into a confirmed, reminded, and recovered appointment without any human scheduling effort.

Why It Matters for SMBs

For home service, professional service, and local businesses, the appointment is the sales funnel. A slow booking response (>30 minutes) drops conversion rates by over 80% in most service verticals. Automating the booking moment and the pre-appointment sequence means the business responds at the speed of software while the owner focuses on delivery.

How It Connects to EmberBots' Existing Systems

Feeds from: Lead Capture Engine — passes lead_name, lead_email, lead_phone, service_requested, lead_source, lead_timestamp via standardized lead payload.
Feeds into: Follow-Up Assistant — on appointment outcome (booked, confirmed, no-show, cancelled), passes appointment_status, appointment_datetime, and follow_up_type.
Feeds into: Ops Back Office — confirmed appointments populate the job queue with service type and duration.
Connected to: AI Front Desk — inbound call or chat booking intent passes directly to this system.

Workflow Logic

Branch A — Lead-Triggered Booking
01
Receive Lead Payload

Input: lead_name, lead_email, lead_phone, service_requested, lead_source, lead_timestamp. Maps service_requested → appointment_type using SERVICE_TO_APPOINTMENT_MAP.

Decision — Booking Mode
→BOOKING_MODE = "auto_offer_link" → proceed to Step 02 (send link)
→BOOKING_MODE = "auto_book_first_available" → proceed to Step 03 (auto-book)
02
Send Booking Link

Query Calendly/Cal.com GET /event_types → retrieve scheduling URL. Send email using EMAIL_TEMPLATE_BOOKING_INVITE from sender_email. Simultaneously send SMS using SMS_TEMPLATE_BOOKING_INVITE. Set lead_status = "booking_link_sent".

Expiry logic: If no booking detected within BOOKING_LINK_EXPIRY_HOURS (default 48h), trigger Follow-Up Assistant with follow_up_type = "no_booking_response".
03
Auto-Book First Available Slot

Query GET /availability for relevant event type, starting from now + BOOKING_BUFFER_MINUTES. Select first slot within BUSINESS_HOURS_START/BUSINESS_HOURS_END and not on BLOCKED_DAYS. Create booking via POST /scheduled_events.

04
Send Confirmation

Send confirmation email (EMAIL_TEMPLATE_CONFIRMATION) + SMS (SMS_TEMPLATE_CONFIRMATION). Create Google Calendar event with attendees: lead_email, owner_email, TEAM_CALENDAR_EMAIL. Set lead_status = "appointment_confirmed".

05
24-Hour Reminder

Send EMAIL_TEMPLATE_REMINDER_24H + SMS_TEMPLATE_REMINDER_24H. SMS includes: "Reply CONFIRM to confirm or RESCHEDULE to change."

Decision — SMS Reply
→"CONFIRM" detected → set attendance_confirmed = true, send confirmation reply
→"RESCHEDULE" detected → send reschedule link, set lead_status = "reschedule_requested"
→No reply within 4H → no action (2H reminder still fires)
06
2-Hour Reminder

Send SMS_TEMPLATE_REMINDER_2H. If attendance_confirmed = false and REQUIRE_CONFIRMATION = true, include softer confirmation prompt.

07
No-Show Detection (T+15min)

Check event status via GET /scheduled_events/{uuid}. If status = active and attendance_confirmed = false and no cancellation webhook received → flag as probable no-show.

Send owner alert email + no-show recovery SMS using SMS_TEMPLATE_NOSHOW. Set lead_status = "no_show". Pass to Follow-Up Assistant: { follow_up_type: "no_show_recovery" }.

08
Cancellation Handling

Triggered by invitee.canceled webhook. Send acknowledgment email + reschedule link. Decline Google Calendar event. Set lead_status = "cancelled".

Decision — Re-Engage
→CANCELLATION_RE_ENGAGE = true → pass to Follow-Up Assistant with follow_up_type = "cancellation_re_engage"
→CANCELLATION_RE_ENGAGE = false → log and end workflow
Branch B — Inbound Email Booking Request

Gmail detects email matching BOOKING_INTENT_KEYWORDS. Extracts sender email and name. Replies using EMAIL_TEMPLATE_BOOKING_INVITE with scheduling link. Resumes at Step 04 once Calendly webhook confirms booking.

Branch C — Self-Booked via Shared Link

Calendly invitee.created webhook fires. Extracts invitee fields. Checks state store for duplicate: if lead exists, updates status; if new, creates lead record and proceeds from Step 04. Sends owner notification: "New booking from [name] — [service] on [date/time]."

System 02

Reputation & Review Engine

Quick Reference
  • TriggersNew review posted on Google or Yelp; scheduled review request sequence after job close
  • Core actionAuto-draft responses; alert team to 1–2 star reviews immediately; send review request via Gmail or SMS post-job
  • BranchingNegative reviews (≤ NEGATIVE_REVIEW_THRESHOLD stars) route to human approval queue before posting; positive reviews auto-post
  • OutputIncreases review volume and average star rating; drives organic lead volume from Google Maps pack
  • Client touchpointOwner approves negative review responses; all positive responses post automatically

System Overview

What Operational Problem It Solves

Local and home service SMBs live and die by their Google star rating. The average owner knows they should respond to reviews and ask for them — but neither happens consistently because both require time and a process they don't have. Negative reviews sit unanswered, satisfied customers who would leave a review never get asked, and the business loses organic visibility to competitors who do manage reviews actively.

Why It Matters for SMBs

Google's local ranking algorithm weighs review volume and recency heavily. A business going from 3.8 to 4.4 stars with 2× the review count can double or triple organic map pack visibility and inbound calls without spending a dollar on ads. For home service businesses, this is the highest-leverage marketing lever available — and the most neglected.

How It Connects to EmberBots' Existing Systems

Feeds from: Ops Back Office — job close event triggers review request sequence with client_name, client_email, client_phone, service_type.
Feeds from: Lead Capture Engine — suppresses review requests to leads still in early funnel stages.
Feeds into: Lead Capture Engine — rising review count improves Google Business Profile visibility, driving more organic leads.
Feeds into: AI Front Desk — negative review alerts surface as context for related inbound calls.

Workflow Logic

Branch A — New Review Detected
01
Poll for New Reviews

Query Google Business Profile: GET accounts/{account}/locations/{location}/reviews?orderBy=updateTime desc&pageSize=10. Compare reviewId values against processed_reviews state store. For each new ID: extract reviewer_display_name, star_rating, comment, create_time. Repeat for Yelp: /businesses/{YELP_BUSINESS_ID}/reviews.

02
Classify Review Sentiment
Decision — Star Rating
→star_rating ≥ POSITIVE_REVIEW_THRESHOLD (default 4) → review_class = "positive"
→star_rating ≤ NEGATIVE_REVIEW_THRESHOLD (default 2) → review_class = "negative"
→Otherwise → review_class = "neutral"
→Positive + empty comment → review_class = "positive_no_comment"
03
Draft Response

Generate response using the appropriate template: RESPONSE_TEMPLATE_POSITIVE, RESPONSE_TEMPLATE_NEUTRAL, or RESPONSE_TEMPLATE_NEGATIVE. For positive reviews: rotate through POSITIVE_RESPONSE_VARIANTS array to avoid duplicate responses across clients and over time. Template supports: {{reviewer_name}}, {{business_name}}, {{service_type}}, {{owner_first_name}}.

04
Route by Classification
Positive / Neutral — Google
→If AUTO_POST_POSITIVE_RESPONSES = true: post immediately via PUT .../reviews/{reviewId}/reply. Accumulate in weekly digest (no immediate email unless NOTIFY_ON_POSITIVE = true).
Positive / Neutral — Yelp
→Cannot auto-post. Email owner with draft + Yelp dashboard link. Subject: "New [X]-star Yelp review from [name] — response draft ready."
Negative Reviews (any platform)
→Immediate SMS alert to owner_phone
→Email to owner_email with review content, drafted response, and approve/edit CTA
→Owner replies "APPROVE" or clicks approval link → post response
→No action within NEGATIVE_REVIEW_RESPONSE_SLA_HOURS (default 24h) → escalation reminder to ESCALATION_EMAIL
05
Log and Track

Write to review_events log: client_id, reviewId, platform, star_rating, review_class, response_posted, response_timestamp, auto_or_manual. Update rolling metrics: current_star_rating, total_review_count, reviews_this_week, avg_response_time_hours.

Branch B — Job Close → Review Request Sequence
01
Receive Job Close Payload

Check SUPPRESSION_LIST: if client_email is present → skip, log suppression. Check if client reviewed within REVIEW_LOOKBACK_DAYS (default 180 days): if yes → skip, log "already reviewed recently."

02
Send First Review Request

At job_close_timestamp + REVIEW_REQUEST_DELAY_HOURS: send SMS using SMS_TEMPLATE_REVIEW_REQUEST including GOOGLE_REVIEW_LINK (short URL). If SEND_REVIEW_REQUEST_EMAIL = true: also send email using EMAIL_TEMPLATE_REVIEW_REQUEST. Max 2 review request messages per job — hard limit.

03
Monitor for Review Submission

Continue polling on REVIEW_POLL_INTERVAL_MINUTES schedule. If new review from matching email or name detected within REVIEW_DETECTION_WINDOW_HOURS (default 96h): set review_request_status = "review_received", cancel pending follow-up, route to Branch A.

04
Send Follow-Up Request (if enabled)

If SEND_REVIEW_FOLLOWUP = true and no review detected after REVIEW_REQUEST_FOLLOWUP_HOURS: send SMS_TEMPLATE_REVIEW_FOLLOWUP. This is the final outreach. Do not send more than 2 review request messages per job under any configuration.

System 03

Payment & Invoice Automation

Quick Reference
  • TriggerJob close event from Ops Back Office; payment webhook from Stripe or Square; scheduled AR aging check
  • Core actionAuto-generate and send invoice on job close; tiered payment reminder sequence; sync payments to QuickBooks; flag overdue accounts
  • BranchingPayment success → QuickBooks sync + optional review trigger; payment failure → auto-retry; overdue → escalation to owner
  • Feeds intoReputation & Review Engine (job paid = trigger for review request); Ops Back Office (payment status updates job record)
  • Client touchpointOwner receives overdue alerts and AR aging report; all invoicing and reminders run autonomously

System Overview

What Operational Problem It Solves

After a job is complete, most small businesses are still manually generating invoices, remembering to follow up on unpaid ones, and hand-entering payments into QuickBooks. Accounts receivable management is one of the most consistent cash flow drains for service SMBs — not because they aren't owed the money, but because collecting it takes time and attention the owner doesn't have. This system automates the entire invoicing and collection lifecycle from job close to QuickBooks reconciliation.

Why It Matters for SMBs

The average small service business carries 15–30 days of unpaid invoices that could be collected faster with consistent, automated follow-up. Automating invoice delivery within minutes of job close — rather than days — reduces payment cycle time materially. Combined with a tiered reminder sequence, collection rates on 30+ day outstanding invoices improve significantly versus manual processes.

How It Connects to EmberBots' Existing Systems

Feeds from: Ops Back Office — job close event with job_id, client_name, client_email, client_phone, service_type, job_amount, job_close_timestamp.
Feeds into: Reputation & Review Engine — payment confirmed event can trigger review request sequence (configurable).
Feeds into: Ops Back Office — payment status updates job record: invoice_status, payment_received_timestamp, payment_amount.

Required Connectors

Workflow Logic

Primary Flow — Job Close to Payment
01
Receive Job Close Payload

Input: job_id, client_name, client_email, client_phone, service_type, job_amount, job_close_timestamp, line_items[] (optional). Validate: job_amount > 0 and client_email is present. If validation fails → alert owner_email and halt.

02
Create or Retrieve Customer Record

Check QuickBooks for existing customer by client_email. If exists: retrieve qb_customer_id. If not: create new customer via QuickBooks API with client_name, client_email, client_phone. Simultaneously check Stripe/Square for existing customer record by email — create if absent.

03
Generate Invoice

Stripe: POST /invoices with customer ID, collection_method: "send_invoice", days_until_due: INVOICE_DUE_DAYS (config). Add line items via POST /invoiceitems. Finalize and send via POST /invoices/{id}/send.

Square: POST /invoices with payment request, due date calculated from INVOICE_DUE_DAYS. Publish via POST /invoices/{id}/publish.

Simultaneously create matching invoice in QuickBooks: POST /invoice with CustomerRef, line items, due date. Write qb_invoice_id and processor_invoice_id to state store linked to job_id.

04
Send Invoice Email

Send email via Gmail using EMAIL_TEMPLATE_INVOICE: includes invoice amount, due date, payment link (from Stripe/Square), itemized summary, business name, and INVOICE_FOOTER_NOTE (config). Set invoice_status = "sent". Log: { event: "invoice_sent", job_id, invoice_id, amount, timestamp }.

05
Payment Reminder Sequence

Three-tier reminder sequence, each only fires if invoice_status != "paid":

R1 (day 3): Email via EMAIL_TEMPLATE_REMINDER_R1 — friendly, informational. "Just a reminder that invoice #{number} for ${amount} is due on {date}."

R2 (day 7): Email + SMS (if SEND_SMS_REMINDERS = true). Slightly firmer tone. Payment link re-included.

R3 (day 14): Email + SMS. Final payment request before escalation. Subject: "Final reminder — invoice #{number} is now past due."

06
Payment Received

Stripe invoice.paid or Square payment.completed webhook fires. Extract: payment_amount, payment_method, payment_timestamp. Mark QB invoice paid: POST /payment with TotalAmt, LinkedTxn referencing QB invoice. Cancel any pending reminder tasks. Set invoice_status = "paid". Log payment event.

Decision — Review Trigger
→If TRIGGER_REVIEW_ON_PAYMENT = true: pass job close event to Reputation & Review Engine with payment_confirmed = true
07
Payment Failed

Stripe invoice.payment_failed or Square payment.failed webhook fires. If AUTO_RETRY_FAILED_PAYMENTS = true: schedule retry at PAYMENT_RETRY_INTERVAL_HOURS (config, default 48h), max MAX_PAYMENT_RETRIES attempts (config, default 2). Send email alert to owner_email: "Payment failed for [client_name] — invoice #{number}." After max retries exhausted: escalate to owner, set invoice_status = "failed".

08
Overdue Escalation (Day 21+)

If invoice_status != "paid" after OVERDUE_ESCALATION_DAYS: send overdue alert to owner_email including invoice amount, days outstanding, and client contact info. SMS alert to owner_phone if SEND_SMS_REMINDERS = true. Add invoice to AR aging bucket in weekly report. Set invoice_status = "overdue". No further automated outreach — owner handles directly.

System Overview

What Operational Problem It Solves

SMB owners running Google Ads face a fundamental visibility problem: they see what they spend but not what they get. Google Ads reports clicks and impressions. What it does not report is whether any of those clicks turned into a phone call, a form fill, a booked appointment, or a paying customer. Without closing that loop, ad spend is managed on vanity metrics — cost-per-click, impressions, CTR — that have no reliable relationship to actual business outcomes. Meanwhile, underperforming campaigns and ad groups bleed budget for weeks because nobody has the time to audit them. This system closes the loop between Google Ads spend and real lead outcomes tracked by EmberBots' Lead Capture Engine, gives the owner a plain-language weekly read on whether their ad dollars are working, and automatically removes or pauses campaigns that are burning budget without producing leads.

Why It Matters for SMBs

For most SMB owners, Google Ads is either their largest or second-largest marketing cost — and the one they understand least. Industry average cost-per-lead for local home service businesses on Google Ads ranges from $30–$120 depending on vertical and market. Without attribution, a business can easily be paying $200+ per lead on specific campaigns while other campaigns deliver at $40, with no visibility into which is which. Identifying and pausing the worst performers alone typically recovers 20–40% of ad spend. For a client spending $2,000/month on Google Ads, that is $400–$800/month recovered — which more than justifies the EmberBots system cost. The weekly digest also gives the owner something they have never had before: a plain-English summary of what their advertising actually produced, without logging into Google Ads.

How It Connects to EmberBots' Existing Systems

Feeds into: Lead Capture Engine — when a lead arrives, this system ensures the lead record is tagged with ad_campaign_id, ad_group_id, keyword, and gclid (Google Click Identifier) from the URL parameters of the landing page; these tags are stored on the lead payload and used to calculate cost-per-lead at the campaign and ad group level.
Feeds from: Lead Capture Engine — lead volume data (count of leads by ad_campaign_id and ad_group_id per time window) is pulled from Lead Capture Engine's state store to calculate CPL per campaign; without this feed, only impression/click data is available.
Feeds into: Reputation & Review Engine — organic keyword visibility data from Google Search Console (impressions, clicks, average position for branded and non-branded terms) is included in the monthly ROI summary with a correlation note when review volume has changed significantly, surfacing the relationship between review activity and organic visibility.
Connected to: AI Front Desk — if the AI Front Desk uses a tracked phone number (via call tracking integration), inbound call attribution can be tied back to the originating Google Ads campaign by matching gclid or UTM parameters; this is an optional enhancement noted in the Configuration Variables.

Workflow Logic

Branch A — Weekly Performance Digest
01
Pull Google Ads Campaign Performance

Issue Google Ads API v18 GoogleAdsService.SearchStream request using GAQL query against resource customers/{GOOGLE_ADS_CUSTOMER_ID}/googleAds:searchStream:

SELECT
  campaign.id,
  campaign.name,
  campaign.status,
  campaign.advertising_channel_type,
  ad_group.id,
  ad_group.name,
  ad_group.status,
  metrics.cost_micros,
  metrics.clicks,
  metrics.impressions,
  metrics.conversions,
  metrics.ctr,
  metrics.average_cpc,
  segments.date
FROM ad_group
WHERE segments.date DURING LAST_7_DAYS
  AND campaign.status != 'REMOVED'
  AND ad_group.status != 'REMOVED'

Convert metrics.cost_micros to dollars: cost_dollars = metrics.cost_micros / 1_000_000. Aggregate by campaign.id: sum cost_dollars, sum metrics.clicks, sum metrics.impressions, sum metrics.conversions. Store result set as campaign_perf[] in working memory; each record: { campaign_id, campaign_name, campaign_status, total_spend, total_clicks, total_impressions, total_conversions, ctr, avg_cpc }.

02
Pull Keyword Performance

Issue second GAQL query for top keywords by spend (top 50, ordered by metrics.cost_micros DESC):

SELECT
  campaign.id,
  campaign.name,
  ad_group.id,
  ad_group.name,
  ad_group_criterion.keyword.text,
  ad_group_criterion.keyword.match_type,
  metrics.cost_micros,
  metrics.clicks,
  metrics.impressions,
  metrics.conversions
FROM keyword_view
WHERE segments.date DURING LAST_7_DAYS
  AND campaign.status != 'REMOVED'
  AND ad_group_criterion.status != 'REMOVED'
ORDER BY metrics.cost_micros DESC
LIMIT 50

Store as keyword_perf[]: { campaign_id, ad_group_id, keyword_text, match_type, cost_dollars, clicks, impressions, conversions }.

03
Pull Lead Volume from Lead Capture Engine

Query Lead Capture Engine state store for leads captured in last 7 days where utm_source = "google" AND (utm_medium = "cpc" OR gclid IS NOT NULL). Group by utm_campaign (maps to campaign.name) and utm_content (maps to ad_group.name). For leads with gclid but no UTM params: attempt campaign match via ad_attribution_log where gclid was recorded at lead capture time. Result: lead_counts_by_campaign[] — { campaign_id_or_name, ad_group_id_or_name, lead_count, lead_ids[] }.

Decision — Zero Attributed Leads
→If Lead Capture Engine returns zero attributed leads for all campaigns: check whether any leads came in with utm_source = "google" but missing gclid — if so, log attribution gap: { event: "attribution_gap_detected", missing_gclid_count, week_ending }
→Flag digest with note: "Some Google leads may be missing UTM parameters — review landing page URL configuration"
04
Calculate Cost-Per-Lead per Campaign

For each entry in campaign_perf[]: match to lead_counts_by_campaign[] by campaign.id or campaign.name. Calculate cpl = total_spend / lead_count (where lead_count > 0). If lead_count = 0: set cpl = null, flag zero_leads = true. Store: { campaign_id, campaign_name, total_spend, lead_count, cpl, zero_leads }.

Compute aggregate totals: total_ad_spend, total_attributed_leads, blended_cpl = total_ad_spend / total_attributed_leads. Compute top keyword CPL: for keywords in keyword_perf[], cross-reference lead attribution records in ad_attribution_log where utm_term or keyword text matches; calculate keyword_cpl where attributable.

05
Pull Google Analytics Session Data (supplement)

GA4 Data API request: POST https://analyticsdata.googleapis.com/v1beta/properties/{GA4_PROPERTY_ID}:runReport. Dimensions: sessionDefaultChannelGrouping, sessionCampaignName, sessionSource, sessionMedium. Metrics: sessions, conversions, bounceRate, averageSessionDuration. DateRange: last 7 days. DimensionFilter: sessionMedium = "cpc" AND sessionSource = "google".

Join to campaign_perf[] on campaign.name vs. sessionCampaignName (case-insensitive). Append ga_sessions, ga_bounce_rate, ga_avg_session_duration to each campaign record.

Decision — Bounce Rate Flag
→If ga_bounce_rate > BOUNCE_RATE_ALERT_THRESHOLD (config, default 0.75) for a campaign: flag high_bounce = true on that campaign record
06
Pull Google Search Console Organic Data

Request: POST https://searchconsole.googleapis.com/webmasters/v3/sites/{SITE_URL}/searchAnalytics/query. Body: { "startDate": "[7_days_ago]", "endDate": "[yesterday]", "dimensions": ["query", "page"], "rowLimit": 25 }. Extract top 10 queries by clicks; record { query, clicks, impressions, ctr, position }.

Compare to prior 7-day baseline stored in search_console_baseline state store. Flag any query where position has changed by more than GSC_POSITION_CHANGE_ALERT (config, default 5 positions) as a notable organic shift. Store organic summary: { branded_impressions, branded_clicks, non_branded_impressions, non_branded_clicks, top_queries[] }.

07
Check Alert and Auto-Pause Thresholds

For each campaign record with cpl IS NOT NULL: if cpl > CPL_ALERT_THRESHOLD (config) AND total_spend >= MIN_SPEND_FOR_ALERT (config, default $50): set alert_flag = true.

For each campaign record with zero_leads = true: if campaign has been running for more than AUTOPAUSE_ZERO_LEADS_DAYS (config) with spend > AUTOPAUSE_MIN_SPEND (config): set autopause_candidate = true. For each campaign record where cpl > AUTOPAUSE_CPL_THRESHOLD (config) AND lead_count < AUTOPAUSE_MIN_LEADS (config): set autopause_candidate = true.

Decision — Branch Routing
→If any alert_flag = true records exist: trigger Branch B immediately (do not wait for digest)
→If any autopause_candidate = true records exist AND AUTOPAUSE_ENABLED = true (config): trigger Branch C immediately
08
Compose and Send Weekly Digest

Compose email using EMAIL_TEMPLATE_WEEKLY_DIGEST. Subject: [BUSINESS_NAME] Ad Performance — Week of [week_start_date].

Section 1 — Summary: Total spend this week: $[total_ad_spend] | Total attributed leads: [total_attributed_leads] | Blended CPL: $[blended_cpl] | vs. prior week: +/-X% (calculated from digest_history state store).

Section 2 — Campaign Breakdown: Table with columns: Campaign Name | Spend | Leads | CPL | Status | 7-Day Trend; rows sorted by spend descending; CPL colored as GOOD (≤ CPL_TARGET), WARNING (≤ CPL_ALERT_THRESHOLD), or HIGH (> CPL_ALERT_THRESHOLD) based on config thresholds.

Section 3 — Top Keywords by Leads: Top 5 keywords ranked by lead_count with CPL per keyword where attributable.

Section 4 — Organic Snapshot: Branded search impressions and clicks this week vs. last week; top 3 non-branded queries by clicks; any significant position changes flagged.

Section 5 — Flags: Any campaigns paused this week (with reason), any attribution gaps detected, any campaigns with high_bounce = true. Footer includes links to Google Ads dashboard and GA4 report (static config URLs).

Send via Gmail API from sender_email to owner_email. Log: { event: "weekly_digest_sent", week_ending, total_spend, total_leads, blended_cpl, campaigns_flagged, campaigns_paused, timestamp }. Write this week's summary to digest_history state store for week-over-week comparison.

Branch B — Real-Time Cost-Per-Lead Alert
Trigger: Step 7 of Branch A sets alert_flag = true on one or more campaigns, OR daily monitoring run detects CPL breach mid-week.
01
Evaluate Alert Severity

For each alert_flag = true campaign: calculate cpl_overage_pct = ((cpl - CPL_ALERT_THRESHOLD) / CPL_ALERT_THRESHOLD) * 100.

Decision — Severity Classification
→If cpl_overage_pct >= 50 (CPL is 1.5× the threshold): alert_severity = "critical"
→If cpl_overage_pct >= 20: alert_severity = "warning"
→Otherwise: alert_severity = "info"

Check alert_suppression state: if an alert for this campaign_id was sent within ALERT_SUPPRESSION_HOURS (config, default 48h), skip email for that campaign to prevent alert fatigue; log suppression.

02
Send Email Alert

Compose using EMAIL_TEMPLATE_CPL_ALERT.

Subject (critical): [BUSINESS_NAME] ALERT: High cost-per-lead on [campaign_name] — $[cpl] vs. $[CPL_ALERT_THRESHOLD] target

Subject (warning): [BUSINESS_NAME] Ad Alert: [campaign_name] CPL above target ($[cpl])

Body: Campaign name | Current CPL | CPL target | Spend this period | Leads this period | Days running above threshold | Recommendation (auto-populated based on severity: "Review ad copy and audience targeting" / "Consider pausing this campaign"). Includes direct deep-link to campaign in Google Ads UI: https://ads.google.com/aw/campaigns?campaignId={campaign_id}.

Send via Gmail API from sender_email to owner_email.

03
Send SMS Alert (if configured)
Decision — SMS Conditions
→If SMS_ALERTS_ENABLED = true AND alert_severity = "critical" AND cpl >= CPL_SMS_ALERT_THRESHOLD (config, default: 2× CPL_ALERT_THRESHOLD): send via Twilio/Sendblue to owner_phone

SMS body: "EmberBots Alert: [campaign_name] CPL is $[cpl] (target $[CPL_ALERT_THRESHOLD]). Check your email for details." Max 160 characters; truncate campaign_name to 30 chars if necessary.

04
Log Alert

Write to alert_log: { event: "cpl_alert_sent", campaign_id, campaign_name, cpl, threshold: CPL_ALERT_THRESHOLD, severity, alert_sent_at, channels: ["email", "sms"] }. Write campaign_id and alert_sent_at to alert_suppression state to enforce suppression window.

Branch C — Underperforming Campaign Auto-Pause
Trigger: Step 7 of Branch A sets autopause_candidate = true on one or more campaigns AND AUTOPAUSE_ENABLED = true.
01
Validate Auto-Pause Conditions

For each autopause_candidate = true campaign, re-verify the condition is still met at time of execution (in case a lead came in since Step 7): re-query Lead Capture Engine for leads attributed to campaign_id in the last AUTOPAUSE_LOOKBACK_DAYS (config, default 7); re-pull current spend via Google Ads API for the same window. Recalculate CPL.

Decision — Condition Still Met?
→Condition no longer met → remove autopause_candidate flag, log: { event: "autopause_condition_cleared", campaign_id }
→AUTOPAUSE_SCOPE = "campaign" (config): pause at the campaign level
→AUTOPAUSE_SCOPE = "ad_group" (config): pause at the ad_group level only (safer — leaves other ad groups in the campaign running)
02
Execute Pause via Google Ads API

Google Ads API v18 mutate request. For campaign-level pause:

POST customers/{GOOGLE_ADS_CUSTOMER_ID}/campaigns:mutate
Body: {
  "operations": [{
    "update": {
      "resourceName": "customers/{GOOGLE_ADS_CUSTOMER_ID}/campaigns/{campaign_id}",
      "status": "PAUSED"
    },
    "updateMask": "status"
  }]
}

For ad group-level pause:

POST customers/{GOOGLE_ADS_CUSTOMER_ID}/adGroups:mutate
Body: {
  "operations": [{
    "update": {
      "resourceName": "customers/{GOOGLE_ADS_CUSTOMER_ID}/adGroups/{ad_group_id}",
      "status": "PAUSED"
    },
    "updateMask": "status"
  }]
}
Decision — API Response
→On API success (HTTP 200 with no partialFailureError): record pause_confirmed = true
→On API failure: do NOT retry automatically; send alert to owner_email + Ember system admin: "Auto-pause failed for [campaign_name] — manual pause required"; log error; do NOT proceed to Step 3
03
Notify Owner

Compose using EMAIL_TEMPLATE_AUTOPAUSE_NOTIFICATION. Subject: [BUSINESS_NAME] — Campaign paused automatically: [campaign_name].

Body includes: Reason for pause (zero leads in [AUTOPAUSE_ZERO_LEADS_DAYS] days / CPL of $[cpl] exceeded auto-pause threshold of $[AUTOPAUSE_CPL_THRESHOLD] with only [lead_count] lead(s)); spend consumed before pause: $[total_spend]; pause scope (campaign-level or ad group-level). Two CTAs: "Resume this campaign" (deep-link to Google Ads) | "Keep paused — I'll review it" (no action required). Note: "This campaign will NOT be re-enabled automatically. You must resume it manually or contact your EmberBots account manager."

Send via Gmail to owner_email. If SMS_ALERTS_ENABLED = true: send SMS to owner_phone: "EmberBots paused your ad campaign '[campaign_name]' — $[total_spend] spent, [lead_count] lead(s). Check email for details."

04
Log Auto-Pause Event

Write to autopause_log: { event: "campaign_auto_paused", campaign_id, campaign_name, ad_group_id (if applicable), pause_scope, reason, spend_at_pause, lead_count_at_pause, cpl_at_pause, pause_timestamp, paused_by: "system" }. Update campaign_perf[] record: set campaign_status = "PAUSED_BY_SYSTEM". Include pause event in next weekly digest under "Flags" section.

Branch D — Monthly ROI Summary
Trigger: First Monday of month at DIGEST_TIME.
01
Pull Full Prior Month Google Ads Data

GAQL query with segments.date DURING LAST_MONTH. Includes all fields from Branch A Step 1 query, plus: campaign.advertising_channel_type, campaign.bidding_strategy_type, metrics.search_impression_share, metrics.search_top_impression_share.

Aggregate totals: monthly_total_spend, monthly_total_clicks, monthly_total_impressions. Compute month-over-month delta vs. monthly_history state store entry for prior month.

02
Pull Monthly Lead Attribution

Query Lead Capture Engine state store for all leads in prior calendar month where utm_source = "google" AND (utm_medium = "cpc" OR gclid IS NOT NULL). Compute: monthly_total_attributed_leads, CPL per campaign for the month, CPL per ad group for the month.

Identify top 3 campaigns by lead count. Identify bottom 3 campaigns by CPL efficiency (highest CPL with meaningful spend, defined as spend > MIN_SPEND_FOR_REPORT, config default $100).

03
Pull Monthly Google Analytics Data

GA4 Data API with dateRange = prior full calendar month. Pull: sessions, conversions, bounce rate by campaign; goal/event completions tagged as LEAD_EVENT_NAME (config, e.g., "generate_lead", "form_submit", "phone_call").

Decision — Attribution Cross-Validation
→Compare GA4 goal completions against Lead Capture Engine lead count; if >20% difference: log delta as attribution_discrepancy and include discrepancy note in report
04
Pull Monthly Search Console Data

Query for prior calendar month. Compute: total organic impressions, total organic clicks, average CTR, average position. Separate branded vs. non-branded queries (branded = queries containing BRAND_KEYWORDS[] config array, e.g., ["[business name]", "[owner name]"]). Compare to prior month baseline; compute month-over-month delta for branded and non-branded impressions.

Decision — Review Correlation
→If INCLUDE_REVIEW_CORRELATION = true (config): pull review count added this month from Reputation & Review Engine state store
→If google_review_count_delta > 5 AND branded_impressions_delta > 0: include correlation note — "Branded search visibility increased [X]% this month — consistent with [N] new Google reviews added"
05
Calculate ROI Summary Figures

monthly_cpl_blended = monthly_total_spend / monthly_total_attributed_leads

estimated_revenue_from_ad_leads = monthly_total_attributed_leads × AVG_JOB_VALUE (config, required field; client provides this value)

estimated_roas = estimated_revenue_from_ad_leads / monthly_total_spend

estimated_monthly_profit_from_ads = estimated_revenue_from_ad_leads - monthly_total_spend

Decision — ROAS Threshold
→If estimated_roas < ROAS_ALERT_THRESHOLD (config, default 2.0): set roas_below_threshold = true

Compare monthly_cpl_blended to prior 3-month CPL average from monthly_history; flag improving or worsening trend.

06
Compose and Send Monthly ROI Summary

Compose using EMAIL_TEMPLATE_MONTHLY_ROI. Subject: [BUSINESS_NAME] Monthly Ad ROI Summary — [Month Year].

Section 1 — Headline Numbers: Total ad spend | Total attributed leads | Blended CPL | Estimated ROAS | Estimated revenue from ad leads.

Section 2 — Campaign Performance Table: Campaign name | Month spend | Leads | CPL | MoM CPL change | Status (Active / Paused / Paused-by-System).

Section 3 — Top Performing Keywords: Top 5 keywords by leads this month; CPL per keyword where attributable; match type.

Section 4 — Organic Search Summary: Branded impressions | Non-branded impressions | Total organic clicks | Avg position | MoM deltas; review correlation note if applicable.

Section 5 — Budget Efficiency: Spend on campaigns that produced 0 leads (waste estimate); spend on campaigns below CPL target (efficient spend); spend on campaigns above CPL target (at-risk spend).

Section 6 — Recommendations (auto-generated from rule engine): if any campaign has been paused for full month → "Consider reviewing or removing [campaign_name] — paused all month with no leads"; if roas_below_threshold = true → "Overall ROAS is [X] — below target of [ROAS_ALERT_THRESHOLD]x. Consider reducing budget on highest-CPL campaigns or reviewing ad copy"; if top keyword CPL < 50% of blended CPL → "Keyword '[keyword_text]' is delivering leads at $[cpl] — [X]% below your average. Consider increasing bids or budget allocation to this keyword."

Section 7 — YTD Trend (if YTD_REPORTING = true config): monthly spend and CPL for each month since system start, displayed as table. Footer includes links to Google Ads, GA4, and Search Console dashboards.

Send via Gmail to owner_email. Write monthly summary to monthly_history state store: { month, total_spend, total_leads, blended_cpl, estimated_roas, campaigns_paused_count }. Log: { event: "monthly_roi_summary_sent", month, total_spend, total_leads, blended_cpl, timestamp }.

Standardized vs. Configured

Standardized — Same for Every Client
  • Google Ads API v18 GAQL query structure for campaign, ad group, and keyword performance pulls
  • metrics.cost_micros to dollars conversion (/ 1_000_000)
  • Lead attribution join logic: matching UTM parameters and gclid from Lead Capture Engine state store to Google Ads campaign and ad group IDs
  • CPL calculation formula: total_spend / lead_count with null guard for zero-lead campaigns
  • Alert severity classification logic (info / warning / critical based on CPL overage %)
  • Alert suppression window enforcement via alert_suppression state
  • Auto-pause eligibility evaluation: zero-lead days check, CPL-above-threshold-with-low-lead-count check, minimum spend guard
  • Google Ads mutate API call structure for campaign and ad group status updates
  • GA4 Data API query structure and session/conversion pull
  • Google Search Console query structure and branded/non-branded separation
  • Week-over-week and month-over-month delta calculations from state store history
  • ROI estimation formula: monthly_total_attributed_leads × AVG_JOB_VALUE
  • Monthly recommendations rule engine (3 canned rule types: paused campaigns, ROAS below threshold, high-performing keyword)
  • Auto-pause irreversibility guarantee: system never re-enables a paused campaign; only the owner can re-enable
  • Error handling: API retry logic (3 attempts, exponential backoff) for all Google API calls; dead-letter queue for failed digest delivery with retry at next hourly interval
  • State store schema: campaign_perf[], digest_history, monthly_history, alert_log, alert_suppression, autopause_log, ad_attribution_log, search_console_baseline
  • Logging schema: all events written to ad_events log with client_id, event_type, timestamp, and relevant payload

Natural Upsell

What the client asks for next
  • Immediate → Lead Capture Engine attribution hardening. At onboarding, this system will almost always surface attribution_gap_count > 0 within the first week: leads arriving from Google without proper UTM parameters or gclid. The fix requires updating landing page URLs and confirming UTM conventions in the Lead Capture Engine. This is a natural, quick-win engagement that keeps the owner involved in the early rollout and demonstrates the system's value before the first weekly digest arrives.
  • Within 30 days → Follow-Up Assistant. Once the owner sees which campaigns produce the most leads, they immediately ask what happens to those leads after capture. If Follow-Up Assistant is not already deployed, the weekly digest's lead volume data makes the ROI case concrete: "You got 18 leads from Google this week. Here's what happened to each of them." Clients who see their CPL drop with good campaign management quickly want to ensure those leads are being worked just as systematically as they're being generated.
  • Within 60 days → Full EmberBots Stack performance consolidation. By Month 2, the client has weekly data from Ad & Lead Gen Manager, Appointment Booking, Reputation & Review Engine, and Payment & Invoice Automation. The natural ask is a unified weekly performance report covering the full funnel: ad spend → leads captured → appointments booked → jobs closed → revenue collected → reviews earned. This consolidated digest is a high-retention artifact and the strongest demonstration of the full EmberBots stack value. It requires no new systems — only a reporting consolidation pass across existing state stores.
Rollout

Implementation Sequencing

The default first deployment for a new client is the Appointment Booking System. It produces the fastest visible result — a lead books an appointment without any manual effort — and immediately demonstrates the EmberBots value proposition. It also feeds into every other system, making it the natural anchor of the full stack.

Exceptions: Start with Reputation & Review Engine instead if the client's Google star rating is below 3.5 — reputation is then the blocker before anything else matters. Start with Payment & Invoice Automation instead if the client has 30+ day-old unpaid invoices or cash flow is the stated priority.
Days 1–30
Foundation — Core Capture & Convert Systems
  • Week 1Deploy AI Front Desk. Configure inbound call capture, lead routing, and initial response templates. Connect to Lead Capture Engine if not yet deployed.
  • Week 1–2Deploy Lead Capture Engine. Connect website contact form, inbound lead sources. Verify lead payload schema is consistent for all downstream systems.
  • Week 2–3Deploy Appointment Booking System. Set BOOKING_MODE, configure Calendly/Cal.com, set business hours and service map. Go live with booking link on website and in AI Front Desk flow.
  • Week 3–4Monitor booking flow. Verify no-show detection timing. Tune message templates based on client feedback. Confirm Follow-Up Assistant is receiving correct handoff payloads.
Days 31–60
Reputation & Back Office
  • Week 5Deploy Follow-Up Assistant. Connect to Appointment Booking and Lead Capture outputs. Configure sequences for booked, no-show, and cold-lead states.
  • Week 5–6Deploy Reputation & Review Engine. Configure Google Business Profile and Yelp connectors. Build review response template variants. Set review request delay to 4 hours post job-close.
  • Week 7Deploy Ops Back Office. Map job lifecycle stages. Connect job close event to both Review Engine and Payment system.
  • Week 7–8Review first 30-day metrics: bookings created, show rate, no-show recovery rate, review volume delta. Tune templates and thresholds where needed.
Days 61–90
Revenue Automation & Full Stack Live
  • Week 9Deploy Payment & Invoice Automation. Connect Stripe or Square. Configure QuickBooks credentials and company ID. Set invoice due days and reminder schedule to match client's existing billing expectations.
  • Week 10Enable TRIGGER_REVIEW_ON_PAYMENT if client prefers payment confirmation as the review trigger vs. job close. Verify dual-trigger suppression is working (no duplicate review requests).
  • Week 10–11Deploy Ad & Lead Gen Manager (if client runs Google Ads). Connect Google Ads, Analytics, and Search Console APIs. Set CPL_TARGET and CPL_ALERT_THRESHOLD based on client's historical CPL. Run first weekly digest as a dry run before enabling auto-pause. Verify UTM attribution tagging is flowing from Lead Capture Engine.
  • Week 11–12Full 8-system stack operational. Confirm all handoff payloads between systems are clean. Enable AUTOPAUSE_ENABLED after owner reviews first two weekly digests and confirms thresholds. Run first monthly ROI summary covering all systems: ad spend, CPL, bookings, show rate, review delta, AR aging, collection rate.
  • Week 12Present 90-day results to client. Open conversation about additional systems or optimization: SMS Rapid Response, Email Nurture, Customer Reactivation based on which bottleneck remains most visible. For Ad & Lead Gen clients, discuss budget reallocation based on CPL data.
