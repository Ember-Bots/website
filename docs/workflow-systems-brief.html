System 01

Appointment Booking System

Quick Reference
  • TriggerNew lead form submission (from Lead Capture Engine) or inbound inquiry via email/SMS
  • Core actionAuto-select available slot, book via Calendly/Cal.com, send confirmation + reminder sequence via Gmail and SMS
  • BranchingNo-show triggers automated rescheduling outreach; cancellation restores slot and fires re-engagement sequence
  • Feeds intoFollow-Up Assistant (marks lead as "appointment booked" or "no-show — re-engage")
  • Client touchpointTeam receives calendar invite + daily booking digest; all booking/reminder logic runs autonomously

System Overview

What Operational Problem It Solves

Most SMBs lose leads between the moment of interest and the moment of a booked appointment. The gap — typically spanning hours to days — is filled with manual scheduling back-and-forth, missed follow-ups, and forgotten confirmations. No-shows compound the problem: without a recovery sequence, a no-show is a dead lead. This system closes that gap by converting a lead signal into a confirmed, reminded, and recovered appointment without any human scheduling effort.

Why It Matters for SMBs

For home service, professional service, and local businesses, the appointment is the sales funnel. A slow booking response (>30 minutes) drops conversion rates by over 80% in most service verticals. Automating the booking moment and the pre-appointment sequence means the business responds at the speed of software while the owner focuses on delivery.

How It Connects to EmberBots' Existing Systems

Feeds from: Lead Capture Engine — passes lead_name, lead_email, lead_phone, service_requested, lead_source, lead_timestamp via standardized lead payload.
Feeds into: Follow-Up Assistant — on appointment outcome (booked, confirmed, no-show, cancelled), passes appointment_status, appointment_datetime, and follow_up_type.
Feeds into: Ops Back Office — confirmed appointments populate the job queue with service type and duration.
Connected to: AI Front Desk — inbound call or chat booking intent passes directly to this system.

Workflow Logic

Branch A — Lead-Triggered Booking
01
Receive Lead Payload

Input: lead_name, lead_email, lead_phone, service_requested, lead_source, lead_timestamp. Maps service_requested → appointment_type using SERVICE_TO_APPOINTMENT_MAP.

Decision — Booking Mode
→BOOKING_MODE = "auto_offer_link" → proceed to Step 02 (send link)
→BOOKING_MODE = "auto_book_first_available" → proceed to Step 03 (auto-book)
02
Send Booking Link

Query Calendly/Cal.com GET /event_types → retrieve scheduling URL. Send email using EMAIL_TEMPLATE_BOOKING_INVITE from sender_email. Simultaneously send SMS using SMS_TEMPLATE_BOOKING_INVITE. Set lead_status = "booking_link_sent".

Expiry logic: If no booking detected within BOOKING_LINK_EXPIRY_HOURS (default 48h), trigger Follow-Up Assistant with follow_up_type = "no_booking_response".
03
Auto-Book First Available Slot

Query GET /availability for relevant event type, starting from now + BOOKING_BUFFER_MINUTES. Select first slot within BUSINESS_HOURS_START/BUSINESS_HOURS_END and not on BLOCKED_DAYS. Create booking via POST /scheduled_events.

04
Send Confirmation

Send confirmation email (EMAIL_TEMPLATE_CONFIRMATION) + SMS (SMS_TEMPLATE_CONFIRMATION). Create Google Calendar event with attendees: lead_email, owner_email, TEAM_CALENDAR_EMAIL. Set lead_status = "appointment_confirmed".

05
24-Hour Reminder

Send EMAIL_TEMPLATE_REMINDER_24H + SMS_TEMPLATE_REMINDER_24H. SMS includes: "Reply CONFIRM to confirm or RESCHEDULE to change."

Decision — SMS Reply
→"CONFIRM" detected → set attendance_confirmed = true, send confirmation reply
→"RESCHEDULE" detected → send reschedule link, set lead_status = "reschedule_requested"
→No reply within 4H → no action (2H reminder still fires)
06
2-Hour Reminder

Send SMS_TEMPLATE_REMINDER_2H. If attendance_confirmed = false and REQUIRE_CONFIRMATION = true, include softer confirmation prompt.

07
No-Show Detection (T+15min)

Check event status via GET /scheduled_events/{uuid}. If status = active and attendance_confirmed = false and no cancellation webhook received → flag as probable no-show.

Send owner alert email + no-show recovery SMS using SMS_TEMPLATE_NOSHOW. Set lead_status = "no_show". Pass to Follow-Up Assistant: { follow_up_type: "no_show_recovery" }.

08
Cancellation Handling

Triggered by invitee.canceled webhook. Send acknowledgment email + reschedule link. Decline Google Calendar event. Set lead_status = "cancelled".

Decision — Re-Engage
→CANCELLATION_RE_ENGAGE = true → pass to Follow-Up Assistant with follow_up_type = "cancellation_re_engage"
→CANCELLATION_RE_ENGAGE = false → log and end workflow
Branch B — Inbound Email Booking Request

Gmail detects email matching BOOKING_INTENT_KEYWORDS. Extracts sender email and name. Replies using EMAIL_TEMPLATE_BOOKING_INVITE with scheduling link. Resumes at Step 04 once Calendly webhook confirms booking.

Branch C — Self-Booked via Shared Link

Calendly invitee.created webhook fires. Extracts invitee fields. Checks state store for duplicate: if lead exists, updates status; if new, creates lead record and proceeds from Step 04. Sends owner notification: "New booking from [name] — [service] on [date/time]."

System 02

Reputation & Review Engine

Quick Reference
  • TriggersNew review posted on Google or Yelp; scheduled review request sequence after job close
  • Core actionAuto-draft responses; alert team to 1–2 star reviews immediately; send review request via Gmail or SMS post-job
  • BranchingNegative reviews (≤ NEGATIVE_REVIEW_THRESHOLD stars) route to human approval queue before posting; positive reviews auto-post
  • OutputIncreases review volume and average star rating; drives organic lead volume from Google Maps pack
  • Client touchpointOwner approves negative review responses; all positive responses post automatically

System Overview

What Operational Problem It Solves

Local and home service SMBs live and die by their Google star rating. The average owner knows they should respond to reviews and ask for them — but neither happens consistently because both require time and a process they don't have. Negative reviews sit unanswered, satisfied customers who would leave a review never get asked, and the business loses organic visibility to competitors who do manage reviews actively.

Why It Matters for SMBs

Google's local ranking algorithm weighs review volume and recency heavily. A business going from 3.8 to 4.4 stars with 2× the review count can double or triple organic map pack visibility and inbound calls without spending a dollar on ads. For home service businesses, this is the highest-leverage marketing lever available — and the most neglected.

How It Connects to EmberBots' Existing Systems

Feeds from: Ops Back Office — job close event triggers review request sequence with client_name, client_email, client_phone, service_type.
Feeds from: Lead Capture Engine — suppresses review requests to leads still in early funnel stages.
Feeds into: Lead Capture Engine — rising review count improves Google Business Profile visibility, driving more organic leads.
Feeds into: AI Front Desk — negative review alerts surface as context for related inbound calls.

Workflow Logic

Branch A — New Review Detected
01
Poll for New Reviews

Query Google Business Profile: GET accounts/{account}/locations/{location}/reviews?orderBy=updateTime desc&pageSize=10. Compare reviewId values against processed_reviews state store. For each new ID: extract reviewer_display_name, star_rating, comment, create_time. Repeat for Yelp: /businesses/{YELP_BUSINESS_ID}/reviews.

02
Classify Review Sentiment
Decision — Star Rating
→star_rating ≥ POSITIVE_REVIEW_THRESHOLD (default 4) → review_class = "positive"
→star_rating ≤ NEGATIVE_REVIEW_THRESHOLD (default 2) → review_class = "negative"
→Otherwise → review_class = "neutral"
→Positive + empty comment → review_class = "positive_no_comment"
03
Draft Response

Generate response using the appropriate template: RESPONSE_TEMPLATE_POSITIVE, RESPONSE_TEMPLATE_NEUTRAL, or RESPONSE_TEMPLATE_NEGATIVE. For positive reviews: rotate through POSITIVE_RESPONSE_VARIANTS array to avoid duplicate responses across clients and over time. Template supports: {{reviewer_name}}, {{business_name}}, {{service_type}}, {{owner_first_name}}.

04
Route by Classification
Positive / Neutral — Google
→If AUTO_POST_POSITIVE_RESPONSES = true: post immediately via PUT .../reviews/{reviewId}/reply. Accumulate in weekly digest (no immediate email unless NOTIFY_ON_POSITIVE = true).
Positive / Neutral — Yelp
→Cannot auto-post. Email owner with draft + Yelp dashboard link. Subject: "New [X]-star Yelp review from [name] — response draft ready."
Negative Reviews (any platform)
→Immediate SMS alert to owner_phone
→Email to owner_email with review content, drafted response, and approve/edit CTA
→Owner replies "APPROVE" or clicks approval link → post response
→No action within NEGATIVE_REVIEW_RESPONSE_SLA_HOURS (default 24h) → escalation reminder to ESCALATION_EMAIL
05
Log and Track

Write to review_events log: client_id, reviewId, platform, star_rating, review_class, response_posted, response_timestamp, auto_or_manual. Update rolling metrics: current_star_rating, total_review_count, reviews_this_week, avg_response_time_hours.

Branch B — Job Close → Review Request Sequence
01
Receive Job Close Payload

Check SUPPRESSION_LIST: if client_email is present → skip, log suppression. Check if client reviewed within REVIEW_LOOKBACK_DAYS (default 180 days): if yes → skip, log "already reviewed recently."

02
Send First Review Request

At job_close_timestamp + REVIEW_REQUEST_DELAY_HOURS: send SMS using SMS_TEMPLATE_REVIEW_REQUEST including GOOGLE_REVIEW_LINK (short URL). If SEND_REVIEW_REQUEST_EMAIL = true: also send email using EMAIL_TEMPLATE_REVIEW_REQUEST. Max 2 review request messages per job — hard limit.

03
Monitor for Review Submission

Continue polling on REVIEW_POLL_INTERVAL_MINUTES schedule. If new review from matching email or name detected within REVIEW_DETECTION_WINDOW_HOURS (default 96h): set review_request_status = "review_received", cancel pending follow-up, route to Branch A.

04
Send Follow-Up Request (if enabled)

If SEND_REVIEW_FOLLOWUP = true and no review detected after REVIEW_REQUEST_FOLLOWUP_HOURS: send SMS_TEMPLATE_REVIEW_FOLLOWUP. This is the final outreach. Do not send more than 2 review request messages per job under any configuration.

System 03

Payment & Invoice Automation

Quick Reference
  • TriggerJob close event from Ops Back Office; payment webhook from Stripe or Square; scheduled AR aging check
  • Core actionAuto-generate and send invoice on job close; tiered payment reminder sequence; sync payments to QuickBooks; flag overdue accounts
  • BranchingPayment success → QuickBooks sync + optional review trigger; payment failure → auto-retry; overdue → escalation to owner
  • Feeds intoReputation & Review Engine (job paid = trigger for review request); Ops Back Office (payment status updates job record)
  • Client touchpointOwner receives overdue alerts and AR aging report; all invoicing and reminders run autonomously

System Overview

What Operational Problem It Solves

After a job is complete, most small businesses are still manually generating invoices, remembering to follow up on unpaid ones, and hand-entering payments into QuickBooks. Accounts receivable management is one of the most consistent cash flow drains for service SMBs — not because they aren't owed the money, but because collecting it takes time and attention the owner doesn't have. This system automates the entire invoicing and collection lifecycle from job close to QuickBooks reconciliation.

Why It Matters for SMBs

The average small service business carries 15–30 days of unpaid invoices that could be collected faster with consistent, automated follow-up. Automating invoice delivery within minutes of job close — rather than days — reduces payment cycle time materially. Combined with a tiered reminder sequence, collection rates on 30+ day outstanding invoices improve significantly versus manual processes.

How It Connects to EmberBots' Existing Systems

Feeds from: Ops Back Office — job close event with job_id, client_name, client_email, client_phone, service_type, job_amount, job_close_timestamp.
Feeds into: Reputation & Review Engine — payment confirmed event can trigger review request sequence (configurable).
Feeds into: Ops Back Office — payment status updates job record: invoice_status, payment_received_timestamp, payment_amount.

Workflow Logic

Primary Flow — Job Close to Payment
01
Receive Job Close Payload

Input: job_id, client_name, client_email, client_phone, service_type, job_amount, job_close_timestamp, line_items[] (optional). Validate: job_amount > 0 and client_email is present. If validation fails → alert owner_email and halt.

02
Create or Retrieve Customer Record

Check QuickBooks for existing customer by client_email. If exists: retrieve qb_customer_id. If not: create new customer via QuickBooks API with client_name, client_email, client_phone. Simultaneously check Stripe/Square for existing customer record by email — create if absent.

03
Generate Invoice

Stripe: POST /invoices with customer ID, collection_method: "send_invoice", days_until_due: INVOICE_DUE_DAYS (config). Add line items via POST /invoiceitems. Finalize and send via POST /invoices/{id}/send.

Square: POST /invoices with payment request, due date calculated from INVOICE_DUE_DAYS. Publish via POST /invoices/{id}/publish.

Simultaneously create matching invoice in QuickBooks: POST /invoice with CustomerRef, line items, due date. Write qb_invoice_id and processor_invoice_id to state store linked to job_id.

04
Send Invoice Email

Send email via Gmail using EMAIL_TEMPLATE_INVOICE: includes invoice amount, due date, payment link (from Stripe/Square), itemized summary, business name, and INVOICE_FOOTER_NOTE (config). Set invoice_status = "sent". Log: { event: "invoice_sent", job_id, invoice_id, amount, timestamp }.

05
Payment Reminder Sequence

Three-tier reminder sequence, each only fires if invoice_status != "paid":

R1 (day 3): Email via EMAIL_TEMPLATE_REMINDER_R1 — friendly, informational. "Just a reminder that invoice #{number} for ${amount} is due on {date}."

R2 (day 7): Email + SMS (if SEND_SMS_REMINDERS = true). Slightly firmer tone. Payment link re-included.

R3 (day 14): Email + SMS. Final payment request before escalation. Subject: "Final reminder — invoice #{number} is now past due."

06
Payment Received

Stripe invoice.paid or Square payment.completed webhook fires. Extract: payment_amount, payment_method, payment_timestamp. Mark QB invoice paid: POST /payment with TotalAmt, LinkedTxn referencing QB invoice. Cancel any pending reminder tasks. Set invoice_status = "paid". Log payment event.

Decision — Review Trigger
→If TRIGGER_REVIEW_ON_PAYMENT = true: pass job close event to Reputation & Review Engine with payment_confirmed = true
07
Payment Failed

Stripe invoice.payment_failed or Square payment.failed webhook fires. If AUTO_RETRY_FAILED_PAYMENTS = true: schedule retry at PAYMENT_RETRY_INTERVAL_HOURS (config, default 48h), max MAX_PAYMENT_RETRIES attempts (config, default 2). Send email alert to owner_email: "Payment failed for [client_name] — invoice #{number}." After max retries exhausted: escalate to owner, set invoice_status = "failed".

08
Overdue Escalation (Day 21+)

If invoice_status != "paid" after OVERDUE_ESCALATION_DAYS: send overdue alert to owner_email including invoice amount, days outstanding, and client contact info. SMS alert to owner_phone if SEND_SMS_REMINDERS = true. Add invoice to AR aging bucket in weekly report. Set invoice_status = "overdue". No further automated outreach — owner handles directly.

Standardized vs. Configured

Standardized — Same for Every Client
  • Stripe and Square invoice creation and webhook handler abstraction
  • QuickBooks customer lookup/creation and invoice sync logic
  • State machine: invoice_status transitions (sent → paid / failed / overdue)
  • Reminder sequence scheduling logic (days-based, cancellation on payment)
  • Failed payment retry logic with exponential backoff
  • AR aging bucket calculation (0-30, 31-60, 61-90, 90+ days)
  • Dual-write pattern: processor invoice + QuickBooks invoice kept in sync
  • Logging schema: all events → invoice_events log
  • Error handling: failed QB sync triggers retry + owner alert
Configured Per Client
  • Stripe or Square credentials (API key, webhook secret)
  • QuickBooks OAuth credentials and company ID
  • Gmail OAuth credentials and sender identity
  • Twilio credentials (if SMS reminders enabled)
  • All timing variables (due days, reminder days, escalation days)
  • All email templates and invoice footer note
  • Payment retry settings
  • Review trigger preference (TRIGGER_REVIEW_ON_PAYMENT)

Natural Upsell

What the client asks for next
  • Immediate → Reputation & Review Engine. Once TRIGGER_REVIEW_ON_PAYMENT = true is enabled, the natural next question is "can you also handle the review request after payment?" This is a one-config-variable conversation.
  • Within 30 days → Customer Reactivation System. With clean payment data in QuickBooks, it becomes easy to identify past clients who haven't returned in 90+ days. The reactivation pitch builds directly on the invoice data already flowing through the system.
  • Within 60 days → Ops Back Office expansion. Clients see job-to-invoice time shrink and often want to connect more of their job management flow — estimates, job costing, technician time tracking — into the same system.
Rollout

Implementation Sequencing

The default first deployment for a new client is the Appointment Booking System. It produces the fastest visible result — a lead books an appointment without any manual effort — and immediately demonstrates the EmberBots value proposition. It also feeds into every other system, making it the natural anchor of the full stack.

Exceptions: Start with Reputation & Review Engine instead if the client's Google star rating is below 3.5 — reputation is then the blocker before anything else matters. Start with Payment & Invoice Automation instead if the client has 30+ day-old unpaid invoices or cash flow is the stated priority.
Days 1–30
Foundation — Core Capture & Convert Systems
  • Week 1Deploy AI Front Desk. Configure inbound call capture, lead routing, and initial response templates. Connect to Lead Capture Engine if not yet deployed.
  • Week 1–2Deploy Lead Capture Engine. Connect website contact form, inbound lead sources. Verify lead payload schema is consistent for all downstream systems.
  • Week 2–3Deploy Appointment Booking System. Set BOOKING_MODE, configure Calendly/Cal.com, set business hours and service map. Go live with booking link on website and in AI Front Desk flow.
  • Week 3–4Monitor booking flow. Verify no-show detection timing. Tune message templates based on client feedback. Confirm Follow-Up Assistant is receiving correct handoff payloads.
Days 31–60
Reputation & Back Office
  • Week 5Deploy Follow-Up Assistant. Connect to Appointment Booking and Lead Capture outputs. Configure sequences for booked, no-show, and cold-lead states.
  • Week 5–6Deploy Reputation & Review Engine. Configure Google Business Profile and Yelp connectors. Build review response template variants. Set review request delay to 4 hours post job-close.
  • Week 7Deploy Ops Back Office. Map job lifecycle stages. Connect job close event to both Review Engine and Payment system.
  • Week 7–8Review first 30-day metrics: bookings created, show rate, no-show recovery rate, review volume delta. Tune templates and thresholds where needed.
Days 61–90
Revenue Automation & Full Stack Live
  • Week 9Deploy Payment & Invoice Automation. Connect Stripe or Square. Configure QuickBooks credentials and company ID. Set invoice due days and reminder schedule to match client's existing billing expectations.
  • Week 10Enable TRIGGER_REVIEW_ON_PAYMENT if client prefers payment confirmation as the review trigger vs. job close. Verify dual-trigger suppression is working (no duplicate review requests).
  • Week 11Full 7-system stack operational. Confirm all handoff payloads between systems are clean. Run first monthly report across all systems: bookings, show rate, review delta, AR aging, collection rate.
  • Week 12Present 90-day results to client. Open conversation about additional systems: SMS Rapid Response, Email Nurture, Customer Reactivation based on which bottleneck remains most visible.
